# 交互设计

> 回答一个核心问题:**我(用户)怎么和这套系统打交道?**
> 一句话答案:**你只跟一个"系统 agent"说人话,它在背后操作系统,再把结果讲给你听。你永远不碰命令。**

---

## 一、你已经在用这套交互方式了

从项目开始到现在,你做的事就是:用自然语言跟 Claude Code 对话 → 它理解你的意图、去操作 → 把结果反馈给你。

**这就是本系统的交互方式。** "系统 agent"不是要新造的东西,而是现成的编码 AI 扮演的角色:

| 场景 | 系统 agent 是谁 |
|---|---|
| 公司电脑 | Claude Code |
| 个人电脑 | Cursor |
| 未来常驻 | OpenClaw |

你换机器/换工具,换的只是"扮演管家的那个 AI",你的交互习惯(说人话)不变。

---

## 二、四层交互链:你的话怎么变成系统的动作

```
┌─────────┐   人话      ┌──────────────┐  调用工具   ┌─────────┐  读写   ┌────────┐
│  你      │ ─────────▶ │  系统 Agent   │ ─────────▶ │ 工具层   │ ─────▶ │ 数据仓  │
│         │ ◀───────── │ (Claude等)   │ ◀───────── │(命令/函数)│ ◀───── │(vault) │
└─────────┘  对话/图     └──────────────┘   返回结果   └─────────┘         └────────┘
  "明天要写周报"          理解→决定调 add_task     add_task(title=..)      存成 .md
  "这周做了啥?"          →组织成人话回你          list(status=done)       读出记录
```

**关键**:你只站在最左边。中间三层都是 agent 的事,你不用知道、也不用学。
就像你现在用 Claude Code,从不需要知道它背后调了 `Read`/`Bash`。

---

## 三、命令行工具是给 agent 用的"手",不是给你用的界面

这是最容易误解的一点,必须说清:

- 我们会做一个**工具层**(如 `add_task`、`list_tasks`、`add_worklog`)。
- 它**不是**让你在终端敲 `add_task "写周报"`。
- 它是 **agent 的手**:当你说"明天要写周报",agent 自己决定去调 `add_task`,你全程只说了那句人话。

工具层写得好,只影响 agent 干活顺不顺;**你的使用成本始终是零学习——只要会说话**。

---

## 四、真正要设计的三样东西

既然 agent 是现成的,要设计的是让**任意 agent 都能变身成你的系统管家**的三样:

### ① 工具层(agent 的手)

一组 agent 能调用的操作,包在 Python 里。已有地基:`system_os/vault.py` 的 CRUD。要补的是更贴近意图的高层操作,例如:

| 工具 | agent 什么时候调 |
|---|---|
| `add_task(...)` | 你说"要做 X" |
| `list_tasks(status=, category=)` | 你问"我有哪些待办" |
| `complete_task(id)` + 生成 worklog | 你说"X 做完了" |
| `daily_review(date)` | 你说"今天回顾一下" |
| `add_mood(...)` + 查对症规则 | 你说"我有点焦虑" |

（这些是逐步做的,不是现在一次写完。）

### ② 操作手册(真正的"agent 定义")

一份**给 AI 读的说明书**(不是代码),让它一读就进入"系统管家"角色。内容包括:
- 你是谁(用户画像的抽象部分)、系统是什么。
- 有哪些工具、什么意图该调什么。
- 行为准则(治理层:低电不做价值判断、先排查生理再谈心理、给框架不替拍板)。
- 反馈风格(简洁、说人话、必要时给清单/表格)。

**同一份手册,三处安放**(这就是跨工具移植的秘密):

| Agent | 手册放哪 |
|---|---|
| Claude Code | `CLAUDE.md` |
| Cursor | `.cursorrules` |
| OpenClaw | 它的系统提示配置 |

内容一致,换工具时只是换"读手册的 AI",手册和工具层不变。

### ③ 反馈形态

**当前决定:先纯对话**(agent 用文字/表格/清单回你),零额外工作量。
将来可选加"简单前端看板"(如每日回顾页面),按需再说。

## 五、子 agent(模块专属管家):现在只预留位置

你举的"情绪模块管家"是个好方向——技术上,Claude Code 支持子 agent(本项目开发时就用过 Explore/Plan 子 agent)。

但按**不过早抽象**的纪律:现在只有工作域一个模块,**现在设计"模块管家框架"几乎必然抽错**(会把工作域的偶然特征当成通用结构)。

**决定**:操作手册里留一段"将来可按模块拆子 agent"的说明,不实作。等出现第 2、3 个模块(如情绪、物品),它们的共性自然显现,再抽子 agent 架构。

将来的形态大概会是:
```
主系统 Agent(总管家)
   ├─ 工作模块子 agent(管 task/planning/decision)
   ├─ 情绪模块子 agent(管 mood + 对症反馈)  ← 未来
   └─ 物品模块子 agent(管 inventory/checklist) ← 未来
```
主 agent 分诊你的话,转给对应子 agent。但这是**以后**的事。

---

## 六、落地顺序(建议)

1. **先做操作手册的最小版**(CLAUDE.md):让 Claude Code 在这台机上就能"变身"系统管家,读你的 work-vault。← 收益最快
2. **补几个高层工具**(add_task / list / complete):让手更好用。
3. 用一段时间,攒出真实痛点,再决定做不做前端、拆不拆子 agent。

---

## 七、一句话记住

**你只跟一个 agent 说人话;agent 靠"工具层"(手)+"操作手册"(灵魂)在背后操作系统;换 Claude/Cursor/OpenClaw 只是换个读同一份手册的 AI;子 agent 以后再拆。**
