# 系统全景

> 这份文档帮你建立整个「个人操作系统」的完整心智模型:各部分怎么拼成一个整体、
> 已经造好了什么、还没造什么、日常使用时一天怎么流转、剩下的路怎么走。
> 不含代码细节,是"地图"而非"零件图"。

---

## 一、一句话:这个系统是什么

一套住在你自己电脑上、隐私优先、跨两台机器的「个人操作系统」,帮你把
**记录、提醒、规划、决策、复盘、情绪调节**这些消耗精力的事务外包给 AI。
数据是你本地的一堆 Markdown 文件,你完全拥有它。

---

## 二、四个视角看同一个系统

系统可以从四个角度理解,它们是**同一个东西的不同切面**,不是四个独立模块。

### 视角 1:四层架构(功能怎么分)—— 用「开饭馆」比喻

| 层 | 作用 | 比喻 | 造好没 |
|---|---|---|---|
| 数据层 | 存放结构化资料 | 食材仓库 | ✅ 已完成 |
| Agent 层 | 干活的 AI 引擎 + 套路 | 厨师 + 菜谱 | ⬜ 未开始 |
| 心跳层 | 定时自动提醒/推送 | 闹钟 | ⬜ 未开始 |
| 治理层 | 给 AI 定的规矩 | 后厨守则 | 🟡 钩子已埋 |

### 视角 2:五种食材(数据怎么存)—— 已全部造好 ✅

```
planning 目标 ──拆解──▶ task 要做的事 ──做完──▶ worklog 做过的事
              ▲                │                     │
              │            decision 怎么选的      攒多了提炼
              └────────────────┴───────▶ rule 验证过的经验
```

| 食材 | 是什么 | 时态 |
|---|---|---|
| task | 要做的一件事(状态会变) | 未来时 |
| worklog | 做过的一件事(只增不改) | 过去时 |
| planning | 想达成的目标/里程碑 | — |
| decision | 一次选择:选了啥、为什么 | — |
| rule | 一条验证过的经验 | — |

（生活域还会有 mood 情绪、inventory 物品、checklist、core_card 内核卡、research 研究，未来再加。）

### 视角 3:八类活(系统能替你干什么)

| # | 活 | 读什么食材 | 归哪层 | 造好没 |
|---|---|---|---|---|
| 1 | 捕获·记一笔 | 写 task/worklog | 数据层 | ⬜ 入口待建 |
| 2 | 提醒·到点推送 | 读 task 的 due | 心跳层 | ⬜ |
| 3 | 核对·场景清点 | 读 checklist+物品 | 心跳层 | ⬜ 生活域 |
| 4 | 规划·拆解排期 | 读 planning+task+精力 | Agent 层 | ⬜ |
| 5 | 决策·理清利弊 | 读选项+rule,写 decision | Agent 层 | 🟡 治理钩子已埋 |
| 6 | 复盘·提炼规则 | 读 worklog,写 rule | Agent 层 | ⬜ |
| 7 | 调研·深度检索 | 写 research | Agent 层 | ⬜ 生活域 |
| 8 | 情绪调节 | 写 mood,读对症规则 | 生活+治理 | ⬜ 生活域 |
| ＋ | 每日回顾(聚合视图) | 读当天全部 | Agent 层 | ⬜ 你已确认要做 |

**铁律**:食材(名词)和活(动词)分开。活的引擎一个都不预先写,用真实需求驱动、一次做一个。

### 视角 4:三层数据边界(隐私怎么守)

```
① 框架层 system-code   —— 零真实数据,可开源      → GitHub(公有/私有)
② 工作域 work-vault    —— 真实工作数据,工作机为源 → GitHub 私有仓,单向同步个人机
③ 个人域 personal-vault—— 生活隐私,仅个人机       → GitHub 私有仓
```

划线依据是**数据可共享度**,不是机器。`domain`(work/personal)由机器身份(MACHINE_ID)自动决定。

## 四、日常使用时,你的一天怎么流转(目标形态)

这是系统**全部做完后**的样子,让你知道我们在朝什么方向走:

```
早上  打开电脑
      → 系统(规划活)读你的 planning+task+今天精力,给出「今日清单」
      → 心跳提醒:今天有 2 件事到 due

白天  随口记:"提醒我下午三点跟导师开会" → 捕获活建一条 task
      做完一件事 → 一句话记 worklog,系统帮你归档
      纠结要不要接个私活 → 决策活帮你列利弊;若你此刻很累且这是不可逆决定,
        治理层会拦一下:"精力低,这种大决定缓缓"

晚上  → 每日回顾活:自动把今天的 task 变化+worklog+情绪聚合成一页给你看
      → 你记一句今天的心情,若是焦虑,系统回应"焦虑的解法是行动,要不先做最小一步?"

每周  → 复盘活扫过一周 worklog,提炼出 1-2 条 rule 存进规则库
      → 下次做类似决策时,这些 rule 会被翻出来参考
```

现在造好的"五种食材",就是上面每一句话背后要读写的数据。活是把它们串起来的引擎。

## 五、真实数据存放方案(已定意向)

| 仓 | 放哪 | 公私 | 为什么 |
|---|---|---|---|
| ① system-code | GitHub | 公有或先私有 | 框架,将来开源 |
| ② work-vault | GitHub | **私有** | 要 work→personal 同步,git 最合适;私有仓免费无限 |
| ③ personal-vault | GitHub | **私有** | 生活隐私 |

**git 相比网盘的关键价值**:版本历史 + 一键回退。手滑删了半个月记录,git 能退回任意历史时刻,网盘做不到。

## 六、剩下步骤的依赖关系与建议顺序

```
步骤4 接真实数据 ──┬──▶ 步骤5 使用入口(记/查任务)← 最先能"用起来"
（配置翻转）       │
                   └──▶ 步骤6 同步器（两台机）
                              │
步骤7 playbook 引擎 + 首个活（每日规划 / 每日回顾）
       │
步骤8 心跳（定时提醒） ── 步骤9 治理层完整化
       │
以后：生活域（情绪/物品/内核卡）
```

**建议**:接真实数据(4)后**先做使用入口(5)**——它让系统第一次"能开上路",你能真正记任务,收益最直接。之后按你最痛的需求挑下一个活来做。
